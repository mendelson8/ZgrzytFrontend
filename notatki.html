WPF: Contact Manager - Głęboka Analiza

XAML potrafi być przytłaczający. Rozłóżmy go na czynniki pierwsze, żebyś rozumiał każdą linijkę, którą wpisujesz.

1. Ustawienia Okna i Układ (Grid) w XAML

Anatomia Taga <Window> i <Grid>

W pliku PDF jasno sprecyzowano wymiary okna[cite: 85]. Tłumaczymy to na atrybuty w XAML:

  • x:Class - Mówi kompilatorowi: "Hej, interfejs graficzny z tego pliku połącz z logiką w pliku C# o nazwie ContactManager.MainWindow".
  • xmlns - To przestrzenie nazw (XML Namespaces). Pozwalają WPF-owi zrozumieć, czym są tagi takie jak Grid czy Button. Bez nich plik to tylko zwykły tekst.
  • WindowStartupLocation="CenterScreen" - Automatycznie wylicza środek monitora przy uruchomieniu[cite: 84].

Następnie używamy <Grid>. Grid w WPF to niewidzialna szachownica . Atrybut Height="Auto" w wierszu oznacza: Daj mi tylko tyle miejsca na wysokość, ile zajmuje mój element wewnątrz (np. menu). Atrybut Height="*" (gwiazdka) oznacza: Zabierz dla mnie absolutnie całą resztę wolnego miejsca na ekranie[cite: 111, 112].

Plik: MainWindow.xaml (Dokładne tłumaczenie)
<Window x:Class="ContactManager.MainWindow"
        xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
        xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
        Title="Contact Manager"
        Height="600" Width="800"
        MinWidth="500" MinHeight="500"
        WindowStartupLocation="CenterScreen">

    <Grid>
        <!-- Deklarujemy dwa wiersze. Jeden na menu (Auto), drugi na resztę (*) -->
        <Grid.RowDefinitions>
            <RowDefinition Height="Auto"/>
            <RowDefinition Height="*"/>
        </Grid.RowDefinitions>

        <!-- Atrybut Grid.Row="0" nakazuje menu wpaść do pierwszego zadeklarowanego wyżej wiersza -->
        <Menu Grid.Row="0">
            <MenuItem Header="File">
                <MenuItem Header="Exit" Click="MenuItem_Exit" />
            </MenuItem>
        </Menu>

        <!-- Grid.Row="1" oznacza, że zakładki wpadną do wiersza zajmującego całą resztę okna -->
        <TabControl Grid.Row="1">
            <TabItem Header="List"> <ListBox/> </TabItem>
            <TabItem Header="Grid"> <DataGrid/> </TabItem>
        </TabControl>
    </Grid>
</Window>

3. Ładowanie Enumów przez ObjectDataProvider w XAML

Co tu się dzieje pod maską?

W zadaniu dodawania kontaktu musimy przypisać Gender (płeć), która w C# jest zdefiniowana jako enum Gender { Male, Female }. W starych technologiach w C# dodawalibyśmy do ComboBoxa elementy pętlą: comboBox.Items.Add(Gender.Male). W WPF robimy to "czyściej" – wyłącznie w XAML[cite: 371, 372].

  • <Window.Resources> - To taki schowek okna. Obiekty wrzucone tutaj (zdefiniowane przez x:Key) są ładowane do pamięci raz i mogą być używane przez każdą kontrolkę wewnątrz okna[cite: 378, 380].
  • ObjectDataProvider - To narzędzie, które zachowuje się jak kod C# wewnątrz XAML. Mówimy mu: "Weź typ System:Enum i uruchom jego wbudowaną metodę GetValues[cite: 381, 383]. Jako argument podaj nasz typ (local:Gender)".
  • {StaticResource GenderList} - To polecenie nakazuje ComboBoxowi: "Znajdź w schowku (Resources) element o kluczu GenderList i wyciągnij z niego dane".

Czym jest atrybut UpdateSourceTrigger=PropertyChanged przy TextBoxie? Normalnie dane przesyłane są z TextBoxa do kodu C# dopiero, gdy klikniesz poza TextBox (utraci focus). Ten atrybut zmusza XAML do aktualizowania danych przy każdym wciśnięciu klawisza klawiatury[cite: 368, 369].

Plik: AddContactWindow.xaml (Zrozumienie DataProvidera i Bindowania)
<Window.Resources>
    <!-- Deklaracja "Wtyczki" do wyciągania Enumów. Nazywamy ją GenderList (x:Key) -->
    <ObjectDataProvider x:Key="GenderList"
                        MethodName="GetValues"
                        ObjectType="{x:Type TypeName=System:Enum}">
        <ObjectDataProvider.MethodParameters>
            <!-- Przekazanie naszego konkretnego Enuma z C# -->
            <x:Type TypeName="local:Gender"/>
        </ObjectDataProvider.MethodParameters>
    </ObjectDataProvider>
</Window.Resources>

<Grid>
    <!-- Zwróć uwagę na UpdateSourceTrigger - natychmiastowo aktualizuje właściwość 'Name' w obiekcie -->
    <TextBox Text="{Binding Name, UpdateSourceTrigger=PropertyChanged}" />

    <!-- Podpinamy zawartość (ItemsSource) pod zasób ze schowka. Wybór użytkownika zapisze się w 'Gender' -->
    <ComboBox ItemsSource="{Binding Source={StaticResource GenderList}}"
              SelectedItem="{Binding Path=Gender}" />
</Grid>

4. Magia Stylów: Triggery i AlternationIndex w XAML

Jak dynamicznie zmieniać tło wierszy w Liście?

Zadanie wymaga, by element z indeksem 0 miał różowe tło, indeks 1 niebieskie tło, potem znów 2 różowe, itd. (zmienne tło co drugi wiersz) [cite: 39, 61]. Osiągnięcie tego w kodzie C# byłoby niezwykle skomplikowane i zepsułoby się natychmiast po usunięciu jakiegokolwiek kontaktu ze środka listy.

  • AlternationCount="2" wstawione do głównego <ListBox> informuje WPF: Stwórz wewnętrzny licznik elementów i zapętlaj go od 0 do 1 (bo wpisaliśmy 2 stany)[cite: 593].
  • <Style TargetType="ListBoxItem"> definiuje zachowanie KAŻDEGO pojedynczego wiersza w Liście. Mówimy mu: to są instrukcje jak masz wyglądać[cite: 591].
  • <Trigger> - To tzw. "Wyzwalacz". Jest to system "Jeżeli -> To" (If/Then) wbudowany w XAML. Tłumacząc linijkę niżej na ludzki język: "Jeżeli (Property) wewnętrzny indeks naprzemienny (AlternationIndex) przyjmie wartość 0 (Value), to wykonaj Settera. Setter mówi: zmień Tło (Background) na #FFAFC5FF"[cite: 594].
Plik: MainWindow.xaml (Dokładna budowa Style.Triggers)
<!-- Kluczowe: AlternationCount="2" zmusza WPF do liczenia wierszy jako 0, 1, 0, 1... -->
<ListBox ItemsSource="{Binding Contacts}" AlternationCount="2">

    <!-- ItemContainerStyle oznacza: Zastosuj te zasady do OPAKOWANIA każdego elementu -->
    <ListBox.ItemContainerStyle>
        <Style TargetType="ListBoxItem">

            <!-- Triggers to nasza lista warunków (If / Then) -->
            <Style.Triggers>

                <!-- IF (AlternationIndex == 0) -->
                <Trigger Property="ItemsControl.AlternationIndex" Value="0">
                    <!-- THEN: Ustaw Tło (Background) na Różowe -->
                    <Setter Property="Background" Value="#FFAFC5FF" />
                </Trigger>

                <!-- IF (AlternationIndex == 1) -->
                <Trigger Property="ItemsControl.AlternationIndex" Value="1">
                    <!-- THEN: Ustaw Tło (Background) na Niebieskie -->
                    <Setter Property="Background" Value="#FF75A1FF" />
                </Trigger>

            </Style.Triggers>
        </Style>
    </ListBox.ItemContainerStyle>

</ListBox>

5. Zaawansowana Walidacja (IValidation)

Architektura rozszerzalna przez pliki DLL

Wykorzystujemy interfejs IValidation oraz dziedziczymy po ValidationRule, aby aplikacja pozwalała na wstrzykiwanie nowych zasad walidacji bez modyfikowania głównego kodu okna.

Plik: PhoneValidationRule.cs
public interface IValidation
{
    string Name { get; }
    string Description { get; }
}

public class PhoneValidationRule : ValidationRule, IValidation
{
    public string Name => "Format Telefonu";
    public string Description => "Format musi wynosić XXX-XXX-XXX.";

    public override ValidationResult Validate(object value, CultureInfo cultureInfo)
    {
        string phone = value as string;
        if (!System.Text.RegularExpressions.Regex.IsMatch(phone, @"^\d{3}-\d{3}-\d{3}$"))
            return new ValidationResult(false, Description);

        return ValidationResult.ValidResult;
    }
}

7. Konwertery (Ocena na gwiazdki i kolor RGB)

Jak przetłumaczyć liczbę 7.5 na ocenę w gwiazdkach i obliczyć kolor RGB?

W WPF nie możemy wpisać matematyki do XAML-a. Pisząc kod, używamy interfejsu IValueConverter, który pośredniczy pomiędzy daną wejściową (np. ocena w Double) a daną wyjściową dla UI (np. ciąg gwiazdek "★★★☆☆" lub kolor pędzla SolidColorBrush).

  • RatingToStarsConverter: Tworzy napis ze znakiem "★" oraz z dopełnieniem "☆" na podstawie formuły Math.Clamp.
  • AverageRatingToStarsConverter: Oblicza średnią ze zbioru ICollection<Chapter> dla konkretnej książki.
  • RatingToColorConverter: Oblicza dynamicznie kanały Red i Green zgodnie z zadaniem (wartości od 0 do 192).
Plik: Converters.cs (Stwórz taki plik w projekcie)
using System;
using System.Collections.Generic;
using System.Globalization;
using System.Linq;
using System.Windows.Data;
using System.Windows.Media;
using Potter.API;

namespace MiNIPotter
{
    public class RatingToStarsConverter : IValueConverter
    {
        public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
        {
            if (value is double rating)
            {
                int fullStars = (int)Math.Clamp(rating, 0, 10);
                return new string('★', fullStars) + new string('☆', 10 - fullStars);
            }
            return "";
        }

        public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture) => throw new NotImplementedException();
    }

    public class AverageRatingToStarsConverter : IValueConverter
    {
        public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
        {
            if (value is ICollection<Chapter> chapters && chapters.Any())
            {
                double avg = chapters.Average(c => c.Rating);
                int fullStars = (int)Math.Clamp(avg, 0, 10);
                return new string('★', fullStars) + new string('☆', 10 - fullStars);
            }
            return new string('☆', 10);
        }

        public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture) => throw new NotImplementedException();
    }

    public class RatingToColorConverter : IValueConverter
    {
        public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
        {
            if (value is double rating)
            {
                byte r = (byte)((1 - (rating / 10)) * 192);
                byte g = (byte)((rating / 10) * 192);
                byte b = 0;
                return new SolidColorBrush(Color.FromRgb(r, g, b));
            }
            return Brushes.Black;
        }

        public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture) => throw new NotImplementedException();
    }
}

Dodatek: Jak zrobić To-Do List w WPF

Zbudowanie prostej listy zadań to świetny sposób na utrwalenie wiedzy. Wykorzystamy tu dynamiczną kolekcję, bindowanie właściwości IsCompleted pod CheckBoxa oraz magię DataTriggera do przekreślania wykonanych zadań.

9. Projekt: Prosta Lista Zadań (To-Do List)

Krok 1: Model Danych (Zadanie)

Każde zadanie na naszej liście musi mieć swój tekst (Title) oraz status ukończenia (IsCompleted). Ponieważ kliknięcie CheckBoxa zmienia status IsCompleted, a my chcemy, żeby interfejs od razu na to zareagował (np. przekreślając tekst), musimy użyć INotifyPropertyChanged.

Plik: ToDoItem.cs
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class ToDoItem : INotifyPropertyChanged
{
    private string title;
    private bool isCompleted;

    public string Title
    {
        get => title;
        set { title = value; OnPropertyChanged(); }
    }

    public bool IsCompleted
    {
        get => isCompleted;
        set { isCompleted = value; OnPropertyChanged(); } // Tu następuje magia!
    }

    public event PropertyChangedEventHandler PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string propName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propName));
    }
}

Krok 2: Widok (XAML) - CheckBox i Przekreślanie

Nasz interfejs składa się z pola do wpisywania nowego zadania (TextBox) oraz przycisku "Dodaj". Poniżej znajduje się ListBox. Najważniejsza część to DataTemplate. Wiążemy IsChecked CheckBoxa z naszą właściwością IsCompleted. Następnie używamy DataTrigger, który mówi: "Jeśli zadanie jest ukończone, ustaw styl tekstu na przekreślony".

                            <ColumnDefinition Width="*"/>
                            <ColumnDefinition Width="Auto"/>
                        </Grid.ColumnDefinitions>

                </DataTemplate>

Krok 3: Logika pod spodem (Code-behind)

W kodzie C# inicjalizujemy naszą ObservableCollection, ustawiamy DataContext i obsługujemy kliknięcia przycisków dodawania oraz usuwania.

Plik: MainWindow.xaml.cs
using System.Collections.ObjectModel;
using System.Windows;
using System.Windows.Controls;

public partial class MainWindow : Window
{
    // Główna lista obserwowalna przez UI
    public ObservableCollection<ToDoItem> Tasks { get; set; }

    public MainWindow()
    {
        InitializeComponent();
        Tasks = new ObservableCollection<ToDoItem>();

        // Ustawiamy okno jako główne źródło danych dla swoich kontrolek
        DataContext = this;
    }

    // Dodawanie nowego zadania
    private void AddBtn_Click(object sender, RoutedEventArgs e)
    {
        if (!string.IsNullOrWhiteSpace(NewTaskTextBox.Text))
        {
            Tasks.Add(new ToDoItem { Title = NewTaskTextBox.Text, IsCompleted = false });
            NewTaskTextBox.Clear(); // Czyścimy pole po dodaniu
        }
    }

    // Usuwanie wybranego zadania (przez przycisk X na elemencie)
    private void DeleteBtn_Click(object sender, RoutedEventArgs e)
    {
        // W WPF, 'sender' to kliknięty przycisk. Używamy jego DataContextu, 
        // żeby dowiedzieć się, do którego ToDoItem jest przypięty.
        if (sender is Button btn && btn.DataContext is ToDoItem itemToDelete)
        {
            Tasks.Remove(itemToDelete);
        }
    }
}
