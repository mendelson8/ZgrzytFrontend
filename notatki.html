Zadanie Praktyczne: Inventory Manager (Zarządca Magazynu)

Cel zadania: Zbudowanie aplikacji do zarządzania sprzętem firmowym, wzorowanej na systemie "Contact Manager"[cite: 8, 10]. Wymaga to połączenia XAML (widok) oraz C# (logika), z wykorzystaniem bindowania danych, konwerterów, walidacji oraz okien modalnych[cite: 37, 432, 647, 654].

1. Główne Okno i Siatka (Grid)

Aplikacja wymaga paska menu na górze okna[cite: 14]. Reszta okna jest podzielona na zakładki[cite: 15]. Minimalny rozmiar okna to 500x500, a początkowy to 800x600[cite: 13, 85]. Okno musi pojawiać się na środku ekranu[cite: 16, 84].

Wyjaśnienie: Używamy kontrolki <Grid>, aby podzielić okno na wiersze. Pierwszy wiersz (Menu) ustawiamy na Height="Auto", aby zajął tylko tyle miejsca, ile potrzebuje[cite: 111]. Drugi wiersz ustawiamy na Height="*", co oznacza, że zakładki (TabControl) zajmą całą pozostałą dostępną przestrzeń[cite: 112].

Plik: MainWindow.xaml
<Window x:Class="InventoryManager.MainWindow"
        Title="Inventory Manager"
        Height="600" Width="800" MinWidth="500" MinHeight="500"
        WindowStartupLocation="CenterScreen">

    <Grid>
        <Grid.RowDefinitions>
            <RowDefinition Height="Auto"/> <!-- Miejsce na pasek menu -->
            <RowDefinition Height="*"/>    <!-- Miejsce na resztę aplikacji -->
        </Grid.RowDefinitions>

        <!-- Pasek Menu na górze okna [cite: 14] -->
        <Menu Grid.Row="0">
            <MenuItem Header="File">
                <MenuItem Header="Exit" Click="MenuItem_Exit"/> <!-- Zamyka aplikację [cite: 21] -->
            </MenuItem>
            <MenuItem Header="Inventory">
                <MenuItem Header="Add Item" Click="MenuItem_AddItem"/>
            </MenuItem>
        </Menu>

        <!-- Reszta okna podzielona na zakładki [cite: 15] -->
        <TabControl Grid.Row="1">
            <TabItem Header="Item List"> <ListBox x:Name="ItemListBox"/> </TabItem>
            <TabItem Header="Data Grid"> <DataGrid/> </TabItem>
        </TabControl>
    </Grid>
</Window>

2. Klasy Danych i Powiadomienia UI

Wymagane jest przechowywanie danych i natychmiastowe aktualizowanie widoku po ich edycji[cite: 37, 38]. Stosujemy tu klasę zaimplementowaną z interfejsem INotifyPropertyChanged[cite: 703].

Wyjaśnienie: Samo użycie ObservableCollection powiadamia tylko o zmianach w samej kolekcji (np. dodaniu lub usunięciu elementu)[cite: 205]. Jeśli jednak zmienimy właściwość istniejącego obiektu (np. cenę produktu), widok o tym nie wie. Wywołanie OnPropertyChanged() informuje interfejs graficzny o konieczności odświeżenia konkretnego pola.

Plik: Product.cs
using System.ComponentModel;
using System.Runtime.CompilerServices;

public enum Category { Electronics, Furniture, Stationery }

public class Product : INotifyPropertyChanged
{
    private string name;

    public string Name
    {
        get => name;
        set { name = value; OnPropertyChanged(); } // Natychmiastowe powiadomienie widoku [cite: 38]
    }

    public Category ItemCategory { get; set; }

    public event PropertyChangedEventHandler PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string propName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propName));
    }
}

3. Okno Modalne i Bindowanie Enumów

Otwarcie okna dodawania blokuje i "wyszarza" okno główne[cite: 45]. W formularzu, wybór wartości typu enum następuje przez ComboBox[cite: 47].

Wyjaśnienie: Funkcja ShowDialog() wyświetla okno w trybie modalnym, co blokuje interakcję z innymi oknami aplikacji do czasu jego zamknięcia[cite: 432, 433]. Aby załadować wartości Enum (np. Kategorie) wprost do XAML-a, używamy ObjectDataProvider[cite: 377]. Ten obiekt wystawia metody i właściwości na zewnątrz [cite: 381], wywołując metodę GetValues z systemowej klasy Enum[cite: 383].

Plik: MainWindow.xaml.cs (Otwieranie okna)
private void MenuItem_AddItem(object sender, RoutedEventArgs e)
{
    Opacity = 0.5; // "Wyszarzenie" okna głównego [cite: 45, 436]

    var addWindow = new AddProductWindow();
    if (addWindow.ShowDialog() == true) // Blokuje okno i czeka na wynik [cite: 432, 434]
    {
        Products.Add(addWindow.NewProduct);
    }

    Opacity = 1.0;
}
Plik: AddProductWindow.xaml (Bindowanie Enum)
<Window.Resources>
    <!-- Pobiera wartości z Enuma 'Category' za pomocą ObjectDataProvider [cite: 381, 382] -->
    <ObjectDataProvider x:Key="CategoryList" MethodName="GetValues" ObjectType="{x:Type TypeName=System:Enum}">
        <ObjectDataProvider.MethodParameters>
            <x:Type TypeName="local:Category"/>
        </ObjectDataProvider.MethodParameters>
    </ObjectDataProvider>
</Window.Resources>

<Grid>
    <!-- ComboBox wyświetla wartości Enum i binduje wybór do obiektu [cite: 47, 404, 405] -->
    <ComboBox ItemsSource="{Binding Source={StaticResource CategoryList}}"
              SelectedItem="{Binding Path=ItemCategory}" />
</Grid>

4. Wygląd Listy: AlternationIndex i ContextMenu

Parzyste i nieparzyste pozycje na liście mają różny kolor tła[cite: 39]. Kliknięcie prawym przyciskiem myszy na element rozwija menu z opcją usunięcia[cite: 643]. Należy to osiągnąć tak, by funkcja była odporna na sortowanie i usuwanie elementów ze środka[cite: 40].

Wyjaśnienie: Właściwość AlternationIndex pozwala na nakładanie różnych kolorów tła na naprzemienne elementy listy[cite: 593, 594]. Opcja AlternationCount="2" w ListBox włącza ten mechanizm, a styl (Trigger) reaguje na wartości 0 lub 1[cite: 609, 611, 625]. Menu kontekstowe podpinamy wprost do grida w DataTemplate.

Plik: MainWindow.xaml (Style ListBoxa)
<ListBox ItemsSource="{Binding Products}" AlternationCount="2">

    <ListBox.ItemContainerStyle>
        <Style TargetType="ListBoxItem">
            <Style.Triggers>
                <!-- Różne tła dla wierszy parzystych i nieparzystych [cite: 39, 610, 612] -->
                <Trigger Property="ItemsControl.AlternationIndex" Value="0">
                    <Setter Property="Background" Value="#FFAFC5FF" />
                </Trigger>
                <Trigger Property="ItemsControl.AlternationIndex" Value="1">
                    <Setter Property="Background" Value="#FF75A1FF" />
                </Trigger>
            </Style.Triggers>
        </Style>
    </ListBox.ItemContainerStyle>

    <ListBox.ItemTemplate>
        <DataTemplate>
            <Grid>
                <!-- Menu kontekstowe do usuwania elementu [cite: 643, 644] -->
                <Grid.ContextMenu>
                    <ContextMenu>
                        <MenuItem Header="Delete item" Click="MenuItem_Delete"/>
                    </ContextMenu>
                </Grid.ContextMenu>

                <TextBlock Text="{Binding Name}" />
            </Grid>
        </DataTemplate>
    </ListBox.ItemTemplate>

</ListBox>

5. Konwertery (IValueConverter) i Odblokowywanie Zawartości

Zakładka ustawień jest domyślnie zablokowana (ukryta)[cite: 645, 646]. Mechanizm konwerterów służy do blokowania/odblokowywania widoczności tej zawartości[cite: 647].

Wyjaśnienie: W WPF nie można bezpośrednio powiązać zmiennej typu bool (True/False) z właściwością Visibility, ponieważ Visibility używa typu wyliczeniowego (Visible, Collapsed, Hidden). Konwerter implementujący interfejs IValueConverter pośredniczy między tymi typami, tłumacząc np. true na Visibility.Visible.

Plik: BoolToVisibilityConverter.cs
using System.Windows.Data;
using System.Windows;

public class BoolToVisibilityConverter : IValueConverter
{
    public object Convert(object value, Type targetType, object parameter, CultureInfo culture)
    {
        if (value is bool isUnlocked && isUnlocked)
            return Visibility.Visible;

        return Visibility.Collapsed;
    }

    public object ConvertBack(object value, Type targetType, object parameter, CultureInfo culture)
    {
        throw new NotImplementedException();
    }
}

6. System Walidacji i IValidation

Należy zaimplementować reguły sprawdzające poprawność wprowadzanych danych, takie jak format XXX-XXX-XXX[cite: 654, 658]. System musi pozwalać na dodawanie reguł z plików DLL bez rekompilacji, wdrażając klasę bazową ValidationRule oraz interfejs IValidation[cite: 663, 664].

Wyjaśnienie: Rozdzielenie reguł do plików DLL wymaga wspólnego kontraktu (interfejsu)[cite: 671]. Klasa implementująca ValidationRule posiada metodę Validate. Jeśli sprawdzana wartość jest niepoprawna, metoda zwraca obiekt ValidationResult z flagą false i opisem błędu, co uruchamia w XAML szablon Validation.ErrorTemplate[cite: 664, 684, 691, 706].

Plik: SerialNumberValidationRule.cs
// Współdzielony interfejs [cite: 665-670]
public interface IValidation
{
    string Name { get; }
    string Description { get; }
}

// Reguła formatu XXX-XXX [cite: 658, 673, 674]
public class SerialValidationRule : ValidationRule, IValidation
{
    public string Name => "Format Numeru Seryjnego";
    public string Description => "Numer musi być w formacie XXX-XXX.";

    public override ValidationResult Validate(object value, CultureInfo cultureInfo)
    {
        string serial = value as string;

        if (!System.Text.RegularExpressions.Regex.IsMatch(serial, @"^\d{3}-\d{3}$"))
        {
            // Błąd walidacji [cite: 681]
            return new ValidationResult(false, Description);
        }

        return ValidationResult.ValidResult; // Walidacja pomyślna [cite: 691]
    }
}
