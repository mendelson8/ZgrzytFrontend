1. Ustawienia Okna i Podział (Layout)

Po co nam Grid i RowDefinitions?

Wymaganie z PDF mówi jasno: "Pasek menu na górze okna, reszta okna podzielona na zakładki". W WPF nie używamy pozycjonowania pikselowego. Używamy <Grid>, który działa jak niewidzialna tabela.

Dzielimy okno na dwa wiersze:

  • Height="Auto" - Pierwszy wiersz dopasuje swoją wysokość do zawartości (czyli będzie miał dokładnie taką wysokość jak Pasek Menu).
  • Height="*" - Znak gwiazdki to "weź całą resztę wolnego miejsca". Tu wrzucimy zakładki (TabControl).
Plik: MainWindow.xaml (Fragment Głównego Układu)
<Window x:Class="ContactManager.MainWindow"
        Title="Contact Manager"
        Height="600" Width="800"
        MinWidth="500" MinHeight="500"
        WindowStartupLocation="CenterScreen">

    <Grid>
        <!-- Definicja naszej tabeli (2 wiersze) -->
        <Grid.RowDefinitions>
            <RowDefinition Height="Auto"/>
            <RowDefinition Height="*"/>
        </Grid.RowDefinitions>

        <!-- Wiersz 0: Menu -->
        <Menu Grid.Row="0">
            <MenuItem Header="File">
                <MenuItem Header="Exit" Click="MenuItem_Exit" />
            </MenuItem>
        </Menu>

        <!-- Wiersz 1: Zakładki zajmujące resztę ekranu -->
        <TabControl Grid.Row="1">
            <TabItem Header="List"> <ListBox/> </TabItem>
            <TabItem Header="Grid"> <DataGrid/> </TabItem>
        </TabControl>
    </Grid>
</Window>

2. Powiadomienia Widoku (INotifyPropertyChanged)

Dlaczego ObservableCollection nie wystarcza?

To najczęstszy błąd w WPF. ObservableCollection<Contact> mówi interfejsowi graficznemu tylko o tym, że dodano lub usunięto element z listy. Nie ma pojęcia o tym, że edytowałeś nazwisko istniejącego kontaktu w DataGridzie!

Aby WPF wiedział, że zmieniła się zawartość konkretnego obiektu, klasa Contact musi zaimplementować interfejs INotifyPropertyChanged. Kiedy zmieniasz właściwość Surname, klasa "krzyczy" do widoku (wywołując OnPropertyChanged): "Hej, zmieniłem się, odrysuj mnie!".

Plik: Contact.cs
using System.ComponentModel;
using System.Runtime.CompilerServices;

public class Contact : INotifyPropertyChanged
{
    private string name;

    // Właściwość z powiadomieniem
    public string Name
    {
        get { return name; }
        set
        {
            name = value;
            // Informujemy interfejs (np. ListBox), że Name się zmieniło!
            OnPropertyChanged();
        }
    }

    // Standardowa implementacja interfejsu (kopiuj-wklej)
    public event PropertyChangedEventHandler PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string propertyName = null)
    {
        PropertyChanged?.Invoke(this, new PropertyChangedEventArgs(propertyName));
    }
}

3. Magia: ComboBox ze zmiennymi typu Enum

Jak wyciągnąć "Male" i "Female" do listy rozwijanej?

W oknie dodawania kontaktu musimy wybrać płeć (Gender). W C# masz przygotowany typ enum Gender { Male, Female }. Jak wrzucić to do XAML-a bez pisania brzydkiego kodu w C#?

Rozwiązaniem z PDFa jest ObjectDataProvider. To specjalne narzędzie w Window.Resources, które "wyciąga" wszystkie wartości z Enuma i tworzy z nich gotową do użycia tablicę, którą podpinamy pod właściwość ItemsSource naszego ComboBoxa.

Plik: AddContactWindow.xaml (Ładowanie Enuma)
<Window.Resources>
    <!-- Pobiera wartości (GetValues) z typu Enum, a dokładnie z naszego local:Gender -->
    <ObjectDataProvider x:Key="GenderList" MethodName="GetValues" ObjectType="{x:Type TypeName=System:Enum}">
        <ObjectDataProvider.MethodParameters>
            <x:Type TypeName="local:Gender"/>
        </ObjectDataProvider.MethodParameters>
    </ObjectDataProvider>
</Window.Resources>

<Grid>
    <!-- ComboBox z podpiętą listą wartości Enum oraz powiązaniem wyboru z właściwością obiektu -->
    <ComboBox ItemsSource="{Binding Source={StaticResource GenderList}}"
              SelectedItem="{Binding Path=Gender}" />
</Grid>

4. Kolorowanie co drugiego wiersza (AlternationIndex)

Dlaczego używamy Style Triggers?

Zadanie wymaga, by parzyste i nieparzyste wiersze na liście kontaktów miały różne tła. Nie możemy robić tego ręcznie (nadając klasom tło przy dodawaniu elementu w kodzie C#), bo jak usuniemy jeden element ze środka listy, kolory się zepsują.

Rozwiązanie WPF to AlternationCount="2" wstawione do ListBoxa. Mówi to liście: Odliczaj wiersze 0, 1, 0, 1, 0, 1.... Następnie w stylach używamy Triggara (wyzwalacza), który mówi: Jeśli AlternationIndex wynosi 0, zrób tło różowe. Jeśli 1, zrób niebieskie.

Plik: MainWindow.xaml (Wewnątrz ListBox)
<!-- Kluczowe: AlternationCount włącza liczenie 0, 1, 0, 1... -->
<ListBox ItemsSource="{Binding Contacts}" AlternationCount="2">

    <ListBox.ItemContainerStyle>
        <Style TargetType="ListBoxItem">
            <Style.Triggers>
                <!-- Wyzwalacz dla wierszy parzystych (0) -->
                <Trigger Property="ItemsControl.AlternationIndex" Value="0">
                    <Setter Property="Background" Value="#FFAFC5FF" />
                </Trigger>

                <!-- Wyzwalacz dla wierszy nieparzystych (1) -->
                <Trigger Property="ItemsControl.AlternationIndex" Value="1">
                    <Setter Property="Background" Value="#FF75A1FF" />
                </Trigger>
            </Style.Triggers>
        </Style>
    </ListBox.ItemContainerStyle>

</ListBox>

5. Zaawansowana Walidacja (Część domowa)

Dlaczego tworzymy interfejs IValidation?

W zadaniu domowym (Home Part) musisz napisać aplikację w taki sposób, by w przyszłości można było dodawać nowe reguły sprawdzania (np. długości hasła) przez pliki .dll, bez modyfikowania głównego kodu programu.

Aby to się udało, plik DLL i Twój program muszą mówić tym samym językiem. Służy do tego interfejs IValidation. Każda nowa reguła walidacji musi go wdrożyć (posiadać Nazwę i Opis błędu) oraz dziedziczyć po bazowej klasie WPF ValidationRule.

Plik: IValidation.cs i PhoneNumberRule.cs
// KROK 1: Współdzielony interfejs, który będzie w pliku DLL
public interface IValidation
{
    string Name { get; }
    string Description { get; }
}

// KROK 2: Przykładowa reguła (Dziedziczy z ValidationRule i z IValidation)
public class PhoneValidationRule : ValidationRule, IValidation
{
    public string Name => "Format Telefonu";
    public string Description => "Numer musi być w formacie XXX-XXX-XXX.";

    // Główna funkcja, którą wywołuje WPF przy sprawdzaniu tekstu
    public override ValidationResult Validate(object value, CultureInfo cultureInfo)
    {
        string phone = value as string;

        // Prosta walidacja używająca wyrażeń regularnych (RegEx)
        if (!System.Text.RegularExpressions.Regex.IsMatch(phone, @"^\d{3}-\d{3}-\d{3}$"))
        {
            // Zwracamy informację o błędzie z naszym opisem!
            return new ValidationResult(false, Description);
        }

        // Jeśli kod dotarł tutaj, wszystko jest OK
        return ValidationResult.ValidResult;
    }
}
